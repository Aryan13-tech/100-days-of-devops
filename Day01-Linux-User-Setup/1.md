# 🚀 Day 01 - Linux User Setup with Non-Interactive Shell

## 📌 Challenge Overview

As part of the **100 Days of DevOps Challenge (KodeKloud Engineer)**,  
the task was to create a user named **kirsty** with a **non-interactive shell** on **App Server 2 (stapp02)**.

This was required to accommodate backup agent tool specifications.

---

## 🖥 Infrastructure Details

- Jump Host: `jumphost`
- Target Server: `stapp02` (App Server 2)
- Login User: `steve`
- Privilege Escalation: `sudo` to root

---

## 🎯 Objective

Create a Linux user with:

- Username: `kirsty`
- Non-interactive shell
- Configured on App Server 2

---

# 🛠 Step-by-Step Implementation

## 1️⃣ Connect to App Server 2 from Jump Host

```bash
ssh steve@stapp02
```

---

## 2️⃣ Switch to Root User

```bash
sudo -i
```

---

## 3️⃣ Check Available Shells

```bash
cat /etc/shells
```

Confirmed availability of:

```
/sbin/nologin
```

---

## 4️⃣ Create User with Non-Interactive Shell

```bash
useradd kirsty -s /sbin/nologin
```

---

## 5️⃣ Verify User Creation

```bash
grep kirsty /etc/passwd
```

Expected Output:

```
kirsty:x:1002:1002::/home/kirsty:/sbin/nologin
```

---

# ✅ Final Result

✔ User `kirsty` successfully created  
✔ Assigned non-interactive shell  
✔ Configured on correct server (stapp02)  
✔ Verified via `/etc/passwd`  
✔ Task passed successfully  

---

# 🔐 Why Use Non-Interactive Shell?

A non-interactive shell prevents a user from logging into the system.

Commonly used for:

- Backup agents
- Application service accounts
- CI/CD users
- Security hardening in production

This improves security by restricting direct system access.

---

# 🧠 Key Learnings

- Creating users using `useradd`
- Assigning specific login shells
- Understanding `/etc/shells`
- Verifying users in `/etc/passwd`
- Applying DevOps security best practices
